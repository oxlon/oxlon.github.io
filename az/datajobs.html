Data Processing and Source Work

This page documents the data collection, validation and processing stage of the Azerbaijan forecasting work. It is not a file-by-file archive; it explains which series families were obtained, which gaps were closed, and which variables must be constructed from data already held rather than downloaded as ready-made public series.

Executive summary

The collection work materially changes the earlier data constraint. Core macroeconomic, external-sector, monetary, fiscal benchmark, banking and sovereign-risk data have been obtained and catalogued from SSC/Azstat, CBAR, the State Customs Committee, the Ministry of Finance, SOFAZ, IMF, FRED, ECB, WRDS/CRSP and Refinitiv/LSEG. The three previously failed FRED rows have also been closed through the FRED API: daily DFF, ECBDFR and ECBMRRFR are now stored in raw/fred/.

The practical conclusion is that public-data collection is no longer the general blocker for 11.1.1 and 11.1.2. The two remaining series issues are processing tasks, not collection tasks: the import-price index is not published as a clean public official series and must be built as a unit-value / partner-price proxy; official core inflation is published by CBAR as point readings in policy material, but not as a long machine-readable time series, so a continuous monthly core series must be reconstructed from CPI components and validated against CBAR point readings.

Closed data gaps

Closed

FRED policy-rate series

DFF, ECBDFR and ECBMRRFR were initially blocked on the fredgraph endpoint. They were then collected through the FRED API as CSV and JSON files; no failed manifest records remain.

Closed

Sovereign spread

WRDS provided identifiers and proxy evidence; the exact sovereign yield history came from Refinitiv/LSEG. The primary spread is now Azerbaijan YTM minus US Treasury CMT through 2026-06.

Clarified

Ministry model package

The sectoral economy, budget and foreign-trade models are present in the Ministry workbooks and methodology documents. The banking-sector behavioural model remains a build task.

Two derived series still to build

Two variables were not confirmed as official long machine-readable public series. That does not stop the work: both can be constructed from data already collected and should be reported as explicit transformations.

$$ UV_t = \frac{\sum_i V_{i,t}}{\sum_i Q_{i,t}},\qquad \pi^{m}_t = \Delta \log(UV_t) $$(D1)

Import-price proxy. A clean official import-price index / unit-value index was not confirmed as a public series. The import-price channel should therefore be built in two ways: first, a unit-value index from customs and SSC trade value/quantity tables; second, a trade-weighted foreign-price indicator using partner CPI/PPI and exchange rates. This is consistent with the Ministry CPI equation, where import inflation is represented through NEER, world food prices, oil and partner prices rather than a standalone official import-price index.

$$ \pi^{core}_t = \pi_t - \pi^{admin}_t - \pi^{seasonal}_t \quad \text{(approximation to the CBAR definition)} $$(D2)

Core / base inflation. CBAR computes and publishes core/base inflation as point readings in policy material, but not as a separate downloadable long CSV/XLSX series. The definition matters: CBAR core excludes administered and seasonal goods/services, not the generic food-energy basket. A continuous monthly indicator must therefore be reconstructed from SSC components and validated against CBAR official point readings.

Processing pipeline

The data work is organised in three layers. The first is source preservation: raw files are not edited and are tracked in the manifest. The second is standardisation: dates, currency, frequency, naming and source metadata are harmonised. The third is model-series construction: derived variables such as sovereign spread, import-price proxy and reconstructed core CPI are calculated separately and documented in reproducible notebooks or scripts.

Conclusion
Public and licensed-source data collection is no longer the general blocker. The remaining work is to build two derived series, move the three handed-over model blocks into the pipeline, and build the banking-sector behavioural module from CBAR/IMF FSI data.
